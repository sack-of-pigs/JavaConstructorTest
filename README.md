# Менеджер Афиши фильмов: Работа с массивами и алгоритмами на Java

Учебный проект, направленный на глубокое закрепление навыков объектно-ориентированного программирования (ООП), управление массивами данных «вручную» (без использования коллекций) и реализацию алгоритмов реверсирования элементов. Все процессы сборки интегрированы с автоматическим конвейером CI.

## 🎯 Цели и задачи проекта
* **Управление массивами**: создание алгоритма динамического расширения массива (`массив = новый массив длиннее на 1`) при добавлении элементов.
* **Алгоритмическая логика (Реверс)**: написание алгоритма вывода фиксированного количества элементов в обратном порядке (от последних добавленных к первым) с учетом лимитов.
* **Использование конструкторов**: применение перегрузки конструкторов для задания гибких настроек лимита вывода (по умолчанию и кастомный).
* **Комплексное Unit-тестирование**: написание автотестов на JUnit 5, проверяющих различные состояния менеджера (пустой, заполненный частично, заполненный свыше лимита).

## 🎬 Бизнес-логика Менеджера Афиши

Класс `AfishaManager` инкапсулирует в себе массив строк (названия фильмов) и лимит вывода последних добавленных элементов.

### 1. Конструкторы менеджера
* **Конструктор по умолчанию**: если лимит не указан, менеджер выставляет лимит вывода в **5 фильмов**.
* **Конструктор с параметром**: позволяет гибко настроить менеджер при создании и задать любой лимит (например, `3` или `7`).

### 2. Реализованные операции (Методы)
* **`add(String movie)`**: Добавление нового фильма в конец списка. При каждом вызове внутренний массив динамически увеличивается на один элемент.
* **`findAll()`**: Вывод всех добавленных фильмов в хронологическом порядке (в порядке добавления).
* **`findLast()`**: Вывод последних добавленных фильмов в обратном порядке. Количество выводимых фильмов ограничено лимитом менеджера:
  * Если фильмов добавлено меньше, чем лимит, метод возвращает все имеющиеся фильмы в обратном порядке.
  * Если фильмов больше лимита, возвращается ровно столько фильмов, сколько указано в лимите.

## 🧪 Тест-дизайн и покрытие автотестами

В рамках тестирования были созданы автотесты, проверяющие поведение менеджера в различных граничных состояниях. 

### Основные сценарии проверок:
* **Для менеджера по умолчанию (лимит 5):**
  * Добавление одного и нескольких фильмов.
  * Вывод `findLast` при количестве фильмов меньше лимита (например, 3 фильма).
  * Вывод `findLast` при количестве фильмов равном лимиту.
* **Для кастомного менеджера (например, с лимитом 3 или 7):**
  * Проверка соблюдения кастомных границ при вызове `findLast`.

Локальная сборка проекта, верификация чистоты кода и выполнение тестов проводятся командой:
```bash
mvn clean verify
```
Проект демонстрирует **100% покрытие кода веток (Branch Coverage)**, а в GitHub Actions успешно проходит сборка с получением зеленого статуса.
